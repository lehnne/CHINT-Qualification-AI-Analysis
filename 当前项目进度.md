## 一、项目背景

**项目名称**：任职资格 AI 分析系统

**平台**：飞书 Aily（支持 Agent + 知识库 + 工作流 + MCP）

**目标**：对任职资格平台中每个人的关键成果和项目经历进行 AI 分析，生成：
1. **个人能力标签** — 按人员 + 岗位序列 + 原职级分类
2. **金标签** — 按岗位序列 + 职级 + 维度，标识区分通过/未通过的关键能力
3. **劣势诊断与精准提问** — 针对个人相对同岗位同职级人员的短板生成面试问题

**项目目的**：准备面试 AI 应用开发产品经理岗位，此项目作为面试作品展示专业能力。

**GitHub 仓库**：https://github.com/lehnne/CHINT-Qualification-AI-Analysis

## 二、术语约定

| 术语 | 说明 |
|------|------|
| **岗位序列** | 即之前说的"职系"。**往后统一叫"岗位序列"**，如技术研发类、产品类、设计类等 |
| **原职级** | 员工当前职级，如 P5、P6 |
| **申报职级** | 员工本次评审申报的目标职级 |
| **金标签** | 同岗位序列 + 同职级中，能区分通过/未通过的关键能力标签 |
| **D 系数** | 区分度系数，D = (通过组标签均分 - 未通过组标签均分) / 未通过组标签均分 |

## 三、个人习惯与偏好

> 以下是我在这项目中形成的习惯和偏好，后续工作请遵循。

### 3.1 命名习惯

- **岗位序列**：统一叫"岗位序列"，不叫"职系"
- **SQL 变量**：分页偏移量用 `@start`，不用 `@Offset`（避免与 SQL 关键字冲突）
- **JSON 字段**：英文 key + 中文 value，如 `position_family_name = "技术研发类"`
- **接口风格**：能合并就合并，倾向单一合并接口而非多个小接口

### 3.2 文档习惯

- SQL 查询语句单独整理一个 md 文件，不混在设计文档里
- Agent Prompt 单独提取为 md 文件，方便复制到飞书 Aily
- 设计文档用 HTML 格式（可视化效果好，适合面试展示）
- 文档中用表格对比，少用长段落

### 3.3 技术偏好

- 数据库是 **SQL Server**，所有查询用 T-SQL 语法
- 分页用 `OFFSET @start ROWS FETCH NEXT @PageSize ROWS ONLY`（SQL Server 2012+）
- 可选筛选条件用 `AND (@Param IS NULL OR Column = @Param)` 模式
- 评审活动 ID（`rn_code`）为**可选参数**，不填则查全部
- 写入接口用 **PUT**（幂等设计，标签数据可重复生成）
- 维度配置作为**可配置元数据**存在数据库中，不写死在 Prompt 里

### 3.4 沟通习惯

- 沟通语言为**中文**
- 喜欢先确认需求再动手，不喜欢直接改
- 修改要保持多个文件的一致性（HTML 设计文档、SQL 手册、Agent Prompt）

## 四、踩过的坑

### 4.1 PostgreSQL → SQL Server 全部重写

**问题**：最初所有 SQL 都按 PostgreSQL 语法写的（UPSERT、LIMIT/OFFSET、pgvector 等），后来才知道任职资格平台用的是 SQL Server。

**教训**：开工前先确认数据库类型。现在 SQL 手册和设计文档中的源数据库查询已全部改为 T-SQL，但 AI 分析系统自有数据库（PostgreSQL）部分保留 PG 语法。

**影响范围**：SQL 手册（3 个查询全部重写）、设计文档 10.8 节 Part A/B/C。

### 4.2 SearchReplace 大块替换失败

**问题**：尝试用 SearchReplace 替换设计文档中约 190 行的 SQL 块时，报 `deserialize params error: missing field`。

**解决**：改用子智能体（Task subagent）来执行大块编辑，或者拆成多个小块分别替换。

**教训**：SearchReplace 的 old_str 不宜过长，尽量控制在 50 行以内。

### 4.3 章节编号漂移

**问题**：在设计文档中插入新的第二章"业务逻辑澄清"后，后续所有章节编号都需要 +1，包括 h2 标题、目录链接、子章节编号。

**教训**：插入新章节时，要同步更新所有引用该章节编号的地方（目录、交叉引用、文案描述）。

### 4.4 GitHub Fine-grained PAT 权限不足

**问题**：首次推送时 403 权限被拒，fine-grained PAT 只有 `metadata=read` 权限，缺少 Contents 写入权限。

**解决**：在 GitHub Settings → Personal Access Tokens 中，将 Repository permissions 的 **Contents** 设为 **Read and Write**。

**教训**：创建 fine-grained PAT 时，除了选对仓库，还要逐一检查每个 permission 的读写级别。

### 4.5 Git Push 被拒（远程已有内容）

**问题**：GitHub 创建仓库时自动初始化了 README，导致首次 push 被 reject（`fetch first`）。

**解决**：`git push --force` 强制推送覆盖远程内容。

### 4.6 职系 → 岗位序列 术语替换

**问题**：用户中途要求将"职系"统一改为"岗位序列"，但设计文档 HTML 中有大量"职系"字样。

**当前状态**：SQL 手册和 Agent Prompt 已全部更新。**设计文档 HTML 中仍有部分"职系"未替换**，后续修改时需注意。

## 六、关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 标签生成时机 | 预生成 + 定时更新 | 成本控制（批量复用上下文）、体验优化（秒级响应）、一致性保障 |
| 标签分组键 | 人员 + 岗位序列 + 原职级 | 同一人可能评审不同岗位序列；原职级代表能力基线 |
| 金标签分组键 | 岗位序列 + 职级 | 按岗位分组样本太小，按岗位序列+职级保证统计显著性 |
| 维度管理 | 可配置元数据（数据库存储） | 新增岗位序列只需更新配置，不改代码 |
| 接口合并 | 单一合并接口（UNION ALL） | Pipeline 不用循环调用，一次拉完 |
| 写入接口用 PUT | 幂等设计 | 标签数据可重复生成，PUT 天然幂等 |
| 字段命名 | 英文 key + 中文 value | 英文 key 便于程序处理，中文 value 便于 AI 理解 |
| 分页变量名 | `start`（非 `offset`） | 避免与 SQL 关键字冲突 |
| 评审活动筛选 | 可选（非必填） | 支持全量查询和按活动查询两种模式 |
| SQL 方言 | SQL Server T-SQL | 任职资格平台使用 SQL Server |

## 八、业务逻辑要点

### 评审周期
- 每两年一次，员工自主申报目标岗位序列 + 职级
- 勾选项目经历和关键成果作为申报材料
- 评审委员会评审 → 通过 / 不通过

### 不同岗位序列的能力维度

| 岗位序列 | 能力维度 |
|---------|---------|
| 技术研发类 | 技术专长、技术创新、客户导向、知识技能 |
| 产品类 | 产品规划、用户洞察、数据分析、商业思维、项目管理 |
| 设计类 | 设计方法论、用户研究、技术实现、审美判断、行业视野 |
| 市场类 | 市场洞察、策略制定、创意能力、渠道管理、效果评估 |
| 管理类 | 团队管理、战略规划、资源协调、绩效驱动、人才培养 |

### 个人能力标签提取逻辑

```
分组键 = (employee_id, position_family_code, original_grade)
```
- 同一个人、同一个岗位序列、同一个原职级的材料合并为一组
- 只取最新一次评审记录，不跨周期合并
- 每条材料归入一个能力维度 → 每个维度下生成 1-3 个标签

### 金标签生成逻辑

```
分组键 = (position_family_code, grade, dimension)
```
- 按岗位序列 + 职级分组（如"技术研发-P6"）
- 组内分为通过组(A)和未通过组(B)
- 在每个维度下逐标签计算区分度 D
- 判定条件：D > 0.25 且 通过组该标签均分 > 65 → 金标签
