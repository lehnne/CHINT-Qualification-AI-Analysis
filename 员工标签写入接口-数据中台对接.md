# 员工能力标签写入接口 · 数据中台对接文档

> 数据中台将 Agent 通过 MCP 工具 `write_employee_tags` 提交的标签 JSON 先存入中间表 `batch_upload_log` 的 `raw_json` 字段，**不解析、不转存**。
> 解析转存到 `employee_capability_tags` 是独立步骤，由数据库定时任务或手动触发。
> 核心难点：输入是嵌套JSON（维度→标签数组），需要展平为数据库行，再批量写入。

## 二、核心逻辑：JSON 展平

输入是嵌套结构，数据库是扁平行，需要做一次**维度展开**：

```
输入 JSON：
  employee_id: "EMP_12345"
  dimension_tags: {
    "技术专长": [tag1, tag2],       ← 2个标签
    "技术创新": [tag3]               ← 1个标签
  }

展平后 = 3 行：
  (EMP_12345, 2025-H1, 技术专长, 复杂架构设计, 88, ...)
  (EMP_12345, 2025-H1, 技术专长, 高并发优化, 85, ...)
  (EMP_12345, 2025-H1, 技术创新, 工具/流程创新, 78, ...)
```

## 三、写入流程：直接存入中间表

数据中台收到 HTTP POST 请求后，**不需要调用存储过程**，只需要做一次简单的 INSERT，将原始 JSON 完整地存入中间表 `batch_upload_log` 的 `raw_json` 字段。

### 处理流程

```
数据中台收到 HTTP POST /api/v1/tags/employee
  ↓
INSERT INTO batch_upload_log (raw_json, status)
VALUES (${request_body}, 'pending');
  ↓
返回成功（code=0）

  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
  以下为独立步骤，由定时任务或手动触发
  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─

解析转存过程读取 batch_upload_log WHERE status='pending'
  ↓
解析 JSON → 展平标签 → 写入 employee_capability_tags
  ↓
更新 batch_upload_log：status=success, employee_count, tag_count
```

### 为什么拆成两步？

| 原因 | 说明 |
|------|------|
| **写入链路要快** | 数据中台只做简单 INSERT，不涉及复杂 JSON 解析，响应时间可控 |
| **失败可重试** | 原始 JSON 已在中间表中，解析失败后不会丢数据，排查问题后重跑解析即可 |
| **解耦** | 写入和解析互不影响，写入方不需要关心下游业务表结构变化 |
| **审计追溯** | 每次写入的原始数据完整保留，可按需回溯对比 |

### 中间表结构

```sql
-- ============================================================
-- 表名：batch_upload_log（中间表/批次原始数据归档）
-- 用途：数据中台直接写入原始JSON，解析转存过程从中读取并转存到业务表
-- ============================================================
CREATE TABLE batch_upload_log (
    id                  BIGSERIAL       PRIMARY KEY,
    raw_json            TEXT            NOT NULL,       -- 完整的原始 JSON 请求体，数据中台直接写入此字段
    status              VARCHAR(20)     NOT NULL        DEFAULT 'pending',  -- pending / processing / success / failed
    batch_id            VARCHAR(50),                   -- 批次ID（解析时从raw_json回填）
    assessment_cycle    VARCHAR(20),                   -- 评审周期（解析时从raw_json回填）
    mode                VARCHAR(20),                   -- initial / incremental（解析时从raw_json回填）
    error_message       TEXT,                          -- 处理失败时的错误信息
    employee_count      INT             DEFAULT 0,     -- 员工数，解析后回填
    tag_count           INT             DEFAULT 0,     -- 标签总数，解析后回填
    created_at          TIMESTAMPTZ     DEFAULT NOW()  -- 写入时间
);

CREATE INDEX idx_batch_log_status  ON batch_upload_log (status);
CREATE INDEX idx_batch_log_batch   ON batch_upload_log (batch_id);
CREATE INDEX idx_batch_log_cycle   ON batch_upload_log (assessment_cycle DESC);
```

## 五、解析转存过程

解析转存是独立步骤，从 `batch_upload_log` 中读取 `status='pending'` 的记录，解析 JSON 并转存到 `employee_capability_tags`。

### 存储过程代码

```sql
-- ============================================================
-- 存储过程：process_batch_upload
-- 功能：从 batch_upload_log 读取原始 JSON，解析转存到 employee_capability_tags
-- 策略：对每个人采用"先删旧标签，再插新标签"，确保重复跑不累积
-- 输入：p_log_id BIGINT — batch_upload_log 的 id（不传则处理所有 pending 记录）
-- 用法：CALL process_batch_upload(null);     -- 处理所有 pending 记录
--       CALL process_batch_upload(123);      -- 处理指定记录
-- ============================================================
CREATE OR REPLACE PROCEDURE process_batch_upload(p_log_id BIGINT DEFAULT NULL)
LANGUAGE plpgsql
AS $$
DECLARE
    v_rec             RECORD;
    v_payload         JSONB;
    v_batch_id        VARCHAR(50);
    v_assessment_cycle VARCHAR(20);
    v_mode             VARCHAR(20);
    v_emp_rec          RECORD;
    v_dim_key          TEXT;
    v_tag_rec          RECORD;
    v_employee_count   INT := 0;
    v_tag_count        INT := 0;
BEGIN
    FOR v_rec IN
        SELECT id, raw_json
        FROM batch_upload_log
        WHERE status = 'pending'
          AND (p_log_id IS NULL OR id = p_log_id)
        ORDER BY id ASC
        FOR UPDATE SKIP LOCKED
    LOOP
        -- 重置计数器
        v_employee_count := 0;
        v_tag_count := 0;

        -- 更新状态为 processing
        UPDATE batch_upload_log SET status = 'processing' WHERE id = v_rec.id;

        BEGIN
            -- 去掉转义的反斜杠（\" → "），再转换为 JSONB
            v_payload := REPLACE(v_rec.raw_json, '\"', '"')::JSONB;

            -- 提取公共字段
            v_batch_id        := v_payload ->> 'batch_id';
            v_assessment_cycle := v_payload ->> 'assessment_cycle';
            v_mode             := v_payload ->> 'mode';

            -- 回填 batch_upload_log 的元数据字段
            UPDATE batch_upload_log
            SET batch_id = v_batch_id,
                assessment_cycle = v_assessment_cycle,
                mode = v_mode
            WHERE id = v_rec.id;

            -- mode = 'initial'：清空整个评审周期的旧数据
            IF v_mode = 'initial' THEN
                DELETE FROM employee_capability_tags
                WHERE assessment_cycle = v_assessment_cycle;
            END IF;

            -- 遍历 tags 数组，解析转存
            FOR v_emp_rec IN
                SELECT * FROM JSONB_TO_RECORDSET(v_payload -> 'tags')
                  AS x(
                      employee_id          VARCHAR(50),
                      employee_name        VARCHAR(50),
                      position_family_code VARCHAR(20),
                      position_family_name VARCHAR(50),
                      original_position    VARCHAR(100),
                      original_grade       VARCHAR(20),
                      target_grade         VARCHAR(20),
                      dimension_tags       JSONB
                  )
            LOOP
                v_employee_count := v_employee_count + 1;

                -- 对每个人，先删掉该评审周期下的旧标签，再插新的
                IF v_mode = 'incremental' THEN
                    DELETE FROM employee_capability_tags
                    WHERE employee_id = v_emp_rec.employee_id
                      AND assessment_cycle = v_assessment_cycle;
                END IF;

                -- 遍历 dimension_tags 的每个维度
                FOR v_dim_key IN
                    SELECT JSONB_OBJECT_KEYS(v_emp_rec.dimension_tags)
                LOOP
                    -- 遍历该维度下的标签数组
                    FOR v_tag_rec IN
                        SELECT * FROM JSONB_TO_RECORDSET(
                            v_emp_rec.dimension_tags -> v_dim_key
                        ) AS y(
                            tag_name         VARCHAR(100),
                            score            SMALLINT,
                            confidence       DECIMAL(3,2),
                            evidence         TEXT,
                            source_materials JSONB
                        )
                    LOOP
                        v_tag_count := v_tag_count + 1;

                        INSERT INTO employee_capability_tags (
                            employee_id,          employee_name,
                            position_family_code, position_family_name,
                            original_position,    original_grade,
                            target_grade,         assessment_cycle,
                            dimension,            tag_name,
                            score,                confidence,
                            evidence,             source_materials,
                            batch_id,             created_at,
                            updated_at
                        ) VALUES (
                            v_emp_rec.employee_id,          v_emp_rec.employee_name,
                            v_emp_rec.position_family_code, v_emp_rec.position_family_name,
                            v_emp_rec.original_position,    v_emp_rec.original_grade,
                            v_emp_rec.target_grade,         v_assessment_cycle,
                            v_dim_key,                      v_tag_rec.tag_name,
                            v_tag_rec.score,                v_tag_rec.confidence,
                            v_tag_rec.evidence,             v_tag_rec.source_materials,
                            v_batch_id,                     NOW(),
                            NOW()
                        );
                    END LOOP;
                END LOOP;
            END LOOP;

            -- 更新归档表，标记成功
            UPDATE batch_upload_log
            SET status = 'success',
                employee_count = v_employee_count,
                tag_count = v_tag_count
            WHERE id = v_rec.id;

        EXCEPTION
            WHEN OTHERS THEN
                UPDATE batch_upload_log
                SET status = 'failed',
                    error_message = SQLERRM
                WHERE id = v_rec.id;
                -- 继续处理下一条 pending 记录，不中断
        END;
    END LOOP;
END;
$$;
```

### 手动触发解析

```sql
-- 处理指定 id 的记录（比如重跑某条失败记录）

```

## 六、写入策略说明

### mode 参数

| mode | 行为 | 适用场景 |
|------|------|---------|
| `initial` | 先清空整个评审周期的所有标签，再全部写入 | 首次全量初始化 |
| `incremental` | 对 **每个人** 先删旧标签，再插新标签，不影响其他人 | 日常增量跑、重跑某个人的标签 |

### 为什么不用 UPSERT 而用"先删后插"？

**之前的问题**：同样的员工、同样的评审周期，如果 Agent 第二次跑时标签变化了，旧标签会残留。比如：

```
第一次跑张三 → 技术专长：[复杂架构设计, 高并发优化]
第二次跑张三 → 技术专长：[复杂架构设计, 微服务架构]
                      ↑ 高并发优化 永远留在表里了
```

**现在的策略**：对每个人 `DELETE + INSERT`，确保每次写入都是该人员该周期的完整快照，不会累积。

### 幂等性

**一样的请求体调用两次，结果一样吗？**
- 写入中间表：不一样，每次 INSERT 都会新增一条记录
- 解析转存后：`employee_capability_tags` 中最终数据一致（幂等）
  - 第一次：DELETE 0 行 → INSERT N 行
  - 第二次：DELETE N 行（上一批的）→ INSERT N 行
- `batch_upload_log` 中会保留多次写入的记录，用于追溯和审计

### 原始数据归档

每次写入都会在 `batch_upload_log` 表中留下一条完整的原始 JSON 记录，主要用途：

| 用途 | 说明 |
|------|------|
| **审计追溯** | 通过 `batch_id` 查询某次写入的原始数据，确认谁在什么时间写了什么 |
| **失败重跑** | 解析失败时原始 JSON 还在，排错后调用 `process_batch_upload(id)` 重跑 |
| **数据对比** | 同一评审周期的两次写入，可以通过 `batch_upload_log` 对比原始数据差异 |
| **问题排查** | 如果 `employee_capability_tags` 数据异常，可以直接从原始 JSON 追溯 |

**查看失败记录**：

```sql
SELECT id, batch_id, error_message, created_at
FROM batch_upload_log
WHERE status = 'failed'
ORDER BY created_at DESC;
```

## 七、数据中台 API 包装建议

MCP 工具 `write_employee_tags` 只有一个参数 `request_body`，类型为 **string**，传入的是整个 JSON 请求体的字符串。数据中台收到 HTTP POST 后，直接将 `request_body` 字符串存入 `batch_upload_log.raw_json` 字段。

```
Agent 分析完成 → 组装完整 JSON
  ↓
调用 write_employee_tags（MCP）
  └─ 参数：request_body = JSON.stringify(完整请求体)  ← 转换成字符串
  ↓
数据中台收到 HTTP POST /api/v1/tags/employee
  Body: request_body 字符串
  ↓
INSERT INTO batch_upload_log (raw_json, status)
VALUES (${request_body}, 'pending');
  ↓
返回成功/失败状态

  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
  定时任务：每 5 分钟执行一次
  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─

CALL process_batch_upload(null);  ← 处理所有 pending 记录
```

**示例配置**（数据中台 SQL 配置）：

```sql
-- 数据中台只需配置这一条 INSERT 语句
INSERT INTO batch_upload_log (raw_json, status)
VALUES (${request_body}, 'pending');
```

**示例代码**（如果数据中台支持自定义逻辑）：

```javascript
function handleWriteEmployeeTags(request) {
    // 1. 直接获取 request_body 字符串
    const bodyStr = request.request_body;   // 类型：string
    
    // 2. 将字符串作为绑定变量存入中间表
    db.execute(
        "INSERT INTO batch_upload_log (raw_json, status) VALUES ($1, 'pending')",
        [bodyStr]  // 直接传字符串，不做任何解析
    );
    
    // 3. 没有报错就是成功
    return { code: 0, message: "success" };
}
```

### 必填字段校验清单

| 字段路径 | 必填 | 说明 |
|---------|------|------|
| `batch_id` | 是 | 批次ID，非空 |
| `assessment_cycle` | 是 | 评审周期，非空 |
| `mode` | 是 | 只能是 `initial` 或 `incremental` |
| `tags` | 是 | 标签数组，不能为空数组 |
| `tags[].employee_id` | 是 | 员工工号，非空 |
| `tags[].employee_name` | 是 | 员工姓名，非空 |
| `tags[].position_family_code` | 是 | 岗位序列代码，非空 |
| `tags[].position_family_name` | 是 | 岗位序列名称，非空 |
| `tags[].dimension_tags` | 是 | 维度标签对象，不能为空对象 |
| `dimension_tags[].tag_name` | 是 | 标签名称，非空 |
| `dimension_tags[].score` | 是 | 0-100 整数 |

## 九、定时调度建议

为了保证解析转存自动执行，建议通过以下方式调度 `process_batch_upload`：

### 方案一：PostgreSQL pg_cron（推荐）

```sql
-- 安装 pg_cron 扩展（需要 superuser）
CREATE EXTENSION IF NOT EXISTS pg_cron;

-- 每 5 分钟执行一次解析转存，处理所有 pending 记录
SELECT cron.schedule('process-batch-upload', '*/5 * * * *', 'CALL process_batch_upload(null)');
```

### 方案二：数据中台定时任务

如果数据中台支持定时任务，配置每 5 分钟执行一次：

```sql
CALL process_batch_upload(null);
```

### 方案三：手动触发

在数据中台的管理后台或数据库客户端手动执行：

```sql
-- 查看待处理的记录
SELECT id, SUBSTRING(raw_json, 1, 80) AS preview, created_at
FROM batch_upload_log
WHERE status = 'pending'
ORDER BY id ASC;

-- 手动触发解析
CALL process_batch_upload(null);
```

## 十、常见问题

### Q1：为什么不用单独 INSERT 每条，而要用解析转存过程？

数据中台只负责简单写入，复杂的 JSON 展平逻辑由解析转存过程处理。这样分工清晰：数据中台不需要理解嵌套 JSON 结构，只需存字符串。

### Q2：mode = 'initial' 会清空旧数据，误操作怎么办？

initial 模式只会在系统上线第一天使用。日常跑的都是 incremental 模式。如果还想加一层保护，可以在 `process_batch_upload` 内部加一个时间限制：

```sql
IF v_mode = 'initial' AND EXTRACT(HOUR FROM NOW()) NOT BETWEEN 2 AND 4 THEN
    RAISE EXCEPTION 'initial 模式只能在凌晨 2:00-4:00 之间执行';
END IF;
```

### Q3：如果只想单独重跑某一个人的标签，其他人的不变？

**支持的**。incremental 模式下，请求体里只塞那一个人的数据即可。解析转存时按 `employee_id` 逐个处理，只删插那一个人的标签。

### Q4：报错 "invalid input syntax for type json: Token \"" 怎么办？

解析转存过程 `process_batch_upload` 内部已经加了 `REPLACE(raw_json, '\"', '"')`，在 `::JSONB` 之前先去掉转义符。如果用的是旧版，重新执行 `CREATE OR REPLACE PROCEDURE` 覆盖即可。

### Q5：数据写入中间表后，多久能查到最终标签？

取决于调度间隔。如果配置了每 5 分钟执行一次 `process_batch_upload`，最长等待 5 分钟。也可以手动调用 `CALL process_batch_upload(null)` 立即处理。

### Q6：解析失败后怎么重试？

```sql
-- 1. 查看失败记录
SELECT id, error_message, created_at
FROM batch_upload_log
WHERE status = 'failed'
ORDER BY created_at DESC;

-- 2. 排查错误原因
SELECT raw_json FROM batch_upload_log WHERE id = 123;

-- 3. 修复问题后，将状态重置为 pending，然后重新解析
UPDATE batch_upload_log SET status = 'pending', error_message = NULL WHERE id = 123;
CALL process_batch_upload(123);
```

### Q7：如果不使用存储过程，数据中台直接用 SQL 怎么写？

```sql
-- 数据中台写入：将原始 JSON 存入中间表
INSERT INTO batch_upload_log (raw_json, status)
VALUES (${request_body}, 'pending');
```

### Q8：数据中台能否不依赖 PostgreSQL 做 JSON 提取？

**可以**。数据中台不需要提取任何 JSON 字段，只需要把整个 `request_body` 字符串传给 `raw_json` 字段即可。`batch_id`、`assessment_cycle`、`mode` 等字段由解析转存过程从 JSON 中提取并回填。
