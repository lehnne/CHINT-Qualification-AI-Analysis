# 员工能力标签写入接口 · 数据中台对接文档

> 数据中台将 Agent 通过 MCP 工具 `write_employee_tags` 提交的标签 JSON 写入 PostgreSQL 的 `employee_capability_tags` 表。
> MCP 工具只有一个参数 `request_body`（字符串），传入整个 JSON 请求体，数据中台收到 HTTP POST 后直接解包处理。
> 核心难点：输入是嵌套JSON（维度→标签数组），需要展平为数据库行，再批量写入。

## 二、核心逻辑：JSON 展平

输入是嵌套结构，数据库是扁平行，需要做一次**维度展开**：

```
输入 JSON：
  employee_id: "EMP_12345"
  dimension_tags: {
    "技术专长": [tag1, tag2],       ← 2个标签
    "技术创新": [tag3]               ← 1个标签
  }

展平后 = 3 行：
  (EMP_12345, 2025-H1, 技术专长, 复杂架构设计, 88, ...)
  (EMP_12345, 2025-H1, 技术专长, 高并发优化, 85, ...)
  (EMP_12345, 2025-H1, 技术创新, 工具/流程创新, 78, ...)
```

## 三、PostgreSQL 存储过程（推荐方案）

写一个 **PostgreSQL 存储过程**，接收整个 JSON 请求体，**先存原始 JSON 到归档表，再解析转存到具体业务表**。**存储过程不返回结果集，适合数据中台的 `executeUpdate()` 调用。**

### 处理流程

```
CALL upsert_employee_tags('{...}')

  1. 将原始 JSON 字符串插入 batch_upload_log（status=pending）
  2. 更新状态为 processing
  3. 解析 JSON，展平标签数据
  4. 写入 employee_capability_tags（先删后插）
  5. 更新 batch_upload_log：status=success, employee_count, tag_count
  6. 如果出错 → 更新 batch_upload_log：status=failed, error_message
```

### 新增表

在执行存储过程之前，需要先建一张归档表：

```sql
-- ============================================================
-- 表名：batch_upload_log（批次原始数据归档）
-- 用途：存储每次写入的完整原始 JSON 请求体，用于审计、追溯和重跑
-- ============================================================
CREATE TABLE batch_upload_log (
    id                  BIGSERIAL       PRIMARY KEY,
    batch_id            VARCHAR(50)     NOT NULL,       -- 批次ID
    assessment_cycle    VARCHAR(20)     NOT NULL,       -- 评审周期
    mode                VARCHAR(20)     NOT NULL,       -- initial / incremental
    raw_json            TEXT            NOT NULL,       -- 完整的原始 JSON 请求体
    status              VARCHAR(20)     NOT NULL DEFAULT 'pending',  -- pending / processing / success / failed
    error_message       TEXT,                           -- 失败时的错误信息
    employee_count      INT             DEFAULT 0,      -- 员工数
    tag_count           INT             DEFAULT 0,      -- 标签总数
    created_at          TIMESTAMPTZ     DEFAULT NOW()
);

CREATE INDEX idx_batch_log_batch   ON batch_upload_log (batch_id);
CREATE INDEX idx_batch_log_cycle   ON batch_upload_log (assessment_cycle DESC);
CREATE INDEX idx_batch_log_status  ON batch_upload_log (status);
```

### 存储过程代码

```sql
-- ============================================================
-- 存储过程：upsert_employee_tags
-- 功能：① 先存原始 JSON 到 batch_upload_log
--       ② 解析转存到 employee_capability_tags
-- 策略：对每个人采用"先删旧标签，再插新标签"，确保重复跑不累积
-- 输入：p_payload TEXT — 完整的 JSON 请求体字符串
-- 用法：CALL upsert_employee_tags('{...}');
-- ============================================================
CREATE OR REPLACE PROCEDURE upsert_employee_tags(p_payload TEXT)
LANGUAGE plpgsql
AS $$
DECLARE
    v_payload         JSONB;
    v_batch_id        VARCHAR(50);
    v_assessment_cycle VARCHAR(20);
    v_mode             VARCHAR(20);
    v_emp_rec          RECORD;
    v_dim_key          TEXT;
    v_tag_rec          RECORD;
    v_log_id           BIGINT;
    v_employee_count   INT := 0;
    v_tag_count        INT := 0;
BEGIN
    -- 第一步：去掉转义的反斜杠（\" → "），再转换为 JSONB
    -- 因为 MCP 传过来的是被 JSON.stringify 转义过的字符串，\" 不是合法 JSON
    v_payload := REPLACE(p_payload, '\"', '"')::JSONB;

    -- 提取公共字段
    v_batch_id        := v_payload ->> 'batch_id';
    v_assessment_cycle := v_payload ->> 'assessment_cycle';
    v_mode             := v_payload ->> 'mode';

    -- ⭐ 第二步：先存原始 JSON 到归档表
    INSERT INTO batch_upload_log (batch_id, assessment_cycle, mode, raw_json, status)
    VALUES (v_batch_id, v_assessment_cycle, v_mode, p_payload, 'processing')
    RETURNING id INTO v_log_id;

    -- mode = 'initial'：清空整个评审周期的旧数据（首次全量初始化）
    IF v_mode = 'initial' THEN
        DELETE FROM employee_capability_tags
        WHERE assessment_cycle = v_assessment_cycle;
    END IF;

    -- 第三步：遍历 tags 数组，解析转存
    FOR v_emp_rec IN
        SELECT * FROM JSONB_TO_RECORDSET(v_payload -> 'tags')
          AS x(
              employee_id          VARCHAR(50),
              employee_name        VARCHAR(50),
              position_family_code VARCHAR(20),
              position_family_name VARCHAR(50),
              original_position    VARCHAR(100),
              original_grade       VARCHAR(20),
              target_grade         VARCHAR(20),
              dimension_tags       JSONB
          )
    LOOP
        v_employee_count := v_employee_count + 1;

        -- 对每个人，先删掉该评审周期下的旧标签，再插新的
        -- 这样重复跑同一个人的标签时，不会累积旧标签
        IF v_mode = 'incremental' THEN
            DELETE FROM employee_capability_tags
            WHERE employee_id = v_emp_rec.employee_id
              AND assessment_cycle = v_assessment_cycle;
        END IF;

        -- 遍历 dimension_tags 的每个维度
        FOR v_dim_key IN
            SELECT JSONB_OBJECT_KEYS(v_emp_rec.dimension_tags)
        LOOP
            -- 遍历该维度下的标签数组
            FOR v_tag_rec IN
                SELECT * FROM JSONB_TO_RECORDSET(
                    v_emp_rec.dimension_tags -> v_dim_key
                ) AS y(
                    tag_name         VARCHAR(100),
                    score            SMALLINT,
                    confidence       DECIMAL(3,2),
                    evidence         TEXT,
                    source_materials JSONB
                )
            LOOP
                v_tag_count := v_tag_count + 1;

                INSERT INTO employee_capability_tags (
                    employee_id,          employee_name,
                    position_family_code, position_family_name,
                    original_position,    original_grade,
                    target_grade,         assessment_cycle,
                    dimension,            tag_name,
                    score,                confidence,
                    evidence,             source_materials,
                    batch_id,             created_at,
                    updated_at
                ) VALUES (
                    v_emp_rec.employee_id,          v_emp_rec.employee_name,
                    v_emp_rec.position_family_code, v_emp_rec.position_family_name,
                    v_emp_rec.original_position,    v_emp_rec.original_grade,
                    v_emp_rec.target_grade,         v_assessment_cycle,
                    v_dim_key,                      v_tag_rec.tag_name,
                    v_tag_rec.score,                v_tag_rec.confidence,
                    v_tag_rec.evidence,             v_tag_rec.source_materials,
                    v_batch_id,                     NOW(),
                    NOW()
                );
            END LOOP;
        END LOOP;
    END LOOP;

    -- 第四步：更新归档表，标记成功
    UPDATE batch_upload_log
    SET status = 'success',
        employee_count = v_employee_count,
        tag_count = v_tag_count
    WHERE id = v_log_id;

EXCEPTION
    WHEN OTHERS THEN
        -- 第五步：如果出错，更新归档表，标记失败
        UPDATE batch_upload_log
        SET status = 'failed',
            error_message = SQLERRM
        WHERE id = v_log_id;
        RAISE;
END;
$$;
```

## 五、写入策略说明

### mode 参数

| mode | 行为 | 适用场景 |
|------|------|---------|
| `initial` | 先清空整个评审周期的所有标签，再全部写入 | 首次全量初始化 |
| `incremental` | 对 **每个人** 先删旧标签，再插新标签，不影响其他人 | 日常增量跑、重跑某个人的标签 |

### 为什么不用 UPSERT 而用"先删后插"？

**之前的问题**：同样的员工、同样的评审周期，如果 Agent 第二次跑时标签变化了，旧标签会残留。比如：

```
第一次跑张三 → 技术专长：[复杂架构设计, 高并发优化]
第二次跑张三 → 技术专长：[复杂架构设计, 微服务架构]
                      ↑ 高并发优化 永远留在表里了
```

**现在的策略**：对每个人 `DELETE + INSERT`，确保每次写入都是该人员该周期的完整快照，不会累积。

### 幂等性

**一样的请求体调用两次，结果一样吗？**
- 是的，完全幂等。
- 第一次：DELETE 0 行 → INSERT N 行
- 第二次：DELETE N 行（上一批的）→ INSERT N 行
- 每次写入都会在 `batch_upload_log` 中新增一条记录，用于追溯和审计

### 原始数据归档

每次写入都会在 `batch_upload_log` 表中留下一条完整的原始 JSON 记录，主要用途：

| 用途 | 说明 |
|------|------|
| **审计追溯** | 通过 `batch_id` 查询某次写入的原始数据，确认谁在什么时间写了什么 |
| **失败重跑** | 处理失败时原始 JSON 还在，排错后可以直接从归档表重新解析 |
| **数据对比** | 同一评审周期的两次写入，可以通过 `batch_upload_log` 对比原始数据差异 |
| **问题排查** | 如果 `employee_capability_tags` 数据异常，可以直接从原始 JSON 追溯 |

**查看失败记录**：

```sql
SELECT * FROM batch_upload_log
WHERE status = 'failed'
ORDER BY created_at DESC;
```

## 六、数据中台 API 包装建议

MCP 工具 `write_employee_tags` 只有一个参数 `request_body`，类型为 **string**，传入的是整个 JSON 请求体的字符串。**数据中台不需要在应用层做 JSON 解析**，直接把字符串传给存储过程的绑定变量，在存储过程内部使用 `::JSONB` 完成转换。流程如下：

```
Agent 分析完成 → 组装完整 JSON
  ↓
调用 write_employee_tags（MCP）
  └─ 参数：request_body = JSON.stringify(完整请求体)  ← 转换成字符串
  ↓
数据中台收到 HTTP POST /api/v1/tags/employee
  Body: request_body 字符串
  ↓
1. 将 request_body 作为绑定变量传给 SQL
    CALL upsert_employee_tags($1)                ← 存储过程，不返回结果集
2. 存储过程内部先 REPLACE 去掉 \" 转义，再 ::JSONB 转换
3. 返回成功/失败状态
```

**关键说明**：
- `request_body` 传过来是 **string** 类型，不是 JSON 对象
- 数据中台**不需要在应用层做 JSON.parse()**，PostgreSQL 的 `::JSONB` 会自动把字符串解析成 JSONB
- 数据中台只需要把 `request_body` 作为绑定变量传入即可
- 使用 `CALL` 调用存储过程，**不返回结果集**，`executeUpdate()` 不会报错

**示例配置**（数据中台 SQL 配置）：

```sql
-- 数据中台只需配置这一条 CALL 语句，将 request_body 字符串直接传入
CALL upsert_employee_tags(${request_body});
```

**示例代码**（如果数据中台支持自定义逻辑）：

```javascript
function handleWriteEmployeeTags(request) {
    // 1. 直接获取 request_body 字符串（不做 JSON.parse）
    const bodyStr = request.request_body;   // 类型：string
    
    // 2. 将字符串作为绑定变量传给存储过程（CALL 不返回结果集）
    db.execute(
        "CALL upsert_employee_tags($1)",
        [bodyStr]  // 直接传字符串，存储过程内部会做 ::JSONB 转换
    );
    
    // 3. 没有报错就是成功
    return { code: 0, message: "success" };
}
```

| 字段路径 | 必填 | 说明 |

| `assessment_cycle` | 是 | 评审周期，非空 |

| `tags[].dimension_tags` | 是 | 维度标签对象，不能为空对象 |
| `dimension_tags[].tag_name` | 是 | 标签名称，非空 |

## 八、常见问题

### Q1：为什么不用单独 INSERT 每条，而要用函数？

函数一次性处理整个 JSON，数据中台只需要调用一次，不需要在应用层写循环逻辑。如果不用函数，数据中台要先解析 JSON → 按维度展开 → 逐条 INSERT，代码更复杂。

### Q2：mode = 'initial' 会清空旧数据，误操作怎么办？

initial 模式只会在系统上线第一天使用。日常跑的都是 incremental 模式，incremental 只删当前批次中每个人的旧标签，不会影响其他人，**即使误操作也不会丢数据**。

如果还想加一层保护，可以在函数内部加一个时间限制：

```sql
-- 只允许在凌晨 2:00-4:00 之间执行 initial 模式
IF v_mode = 'initial' AND EXTRACT(HOUR FROM NOW()) NOT BETWEEN 2 AND 4 THEN
    RAISE EXCEPTION 'initial 模式只能在凌晨 2:00-4:00 之间执行';
END IF;
```

### Q3：如果只想单独重跑某一个人的标签，其他人的不变？

**支持的**。incremental 模式下，函数按 `employee_id` 逐个处理，请求体里只塞那一个人的数据即可：

```json
{
  "batch_id": "BATCH_20250715_0001",
  "assessment_cycle": "2025-H1",
  "mode": "incremental",
  "tags": [
    {
      "employee_id": "EMP_12345",
      "employee_name": "张三",
      ...
    }
  ]
}
```

函数只会 DELETE + INSERT 张三的数据，李四、王五的标签不受影响。

### Q4：报错 "invalid input syntax for type json: Token \"" 怎么办？

这是 MCP 工具将 `request_body` 序列化时，JSON 字符串内部的引号被转义成了 `\"`，导致 PostgreSQL 的 `::JSONB` 不认识。

**修复方法**：函数内部已经加了 `REPLACE(p_payload, '\"', '"')`，在 `::JSONB` 之前先去掉转义符。如果你用的是旧版函数，重新执行一次 `CREATE OR REPLACE FUNCTION` 覆盖即可。

**为什么会这样？**
```
MCP 传参时序列化过程：
  request_body 的原始值 → "{"batch_id": "BATCH_001"}"
  被 JSON.stringify 后  → "{\"batch_id\": \"BATCH_001\"}"
  PostgreSQL 收到后      → 字符串含 \"，::JSONB 不认

函数修复后：
  REPLACE(\" → ")       → "{"batch_id": "BATCH_001"}"
  ::JSONB               → 解析成功
```

### Q5：报错 "A result was returned when none was expected" 怎么办？

这是数据中台用 `executeUpdate()` 执行了一个返回结果集的 SQL。旧版用的是 `SELECT * FROM upsert_employee_tags(...)`（函数返回 TABLE），导致 JDBC 报错。

**修复方法**：改为存储过程 + `CALL` 调用。将 `CREATE FUNCTION ... RETURNS TABLE` 改为 `CREATE PROCEDURE`，数据中台配置从 `SELECT * FROM upsert_employee_tags(?)` 改为 `CALL upsert_employee_tags(?)`。

**存储过程 vs 函数的区别**：

| 对比项 | 函数（FUNCTION） | 存储过程（PROCEDURE） |
|--------|-----------------|---------------------|
| 返回结果 | 必须返回，可以是 TABLE/VOID | 不返回结果集 |
| 调用方式 | `SELECT func()` | `CALL proc()` |
| JDBC 调用 | `executeQuery()` | `executeUpdate()` |
| 适用场景 | 查询、计算 | 写入、更新、DDL |

### Q6：如果不使用存储过程，数据中台直接用 SQL 怎么写？

在数据中台应用层做循环，先删后插：

```sql
-- 对每个人，先删旧标签
DELETE FROM employee_capability_tags
WHERE employee_id = :employee_id
  AND assessment_cycle = :assessment_cycle;

-- 再逐条 INSERT 新标签
INSERT INTO employee_capability_tags (...)
VALUES (...);
```

### Q7：存储过程处理失败后怎么办？

**不用慌**。存储过程有异常处理机制，失败时会在 `batch_upload_log` 中记录原始 JSON 和错误信息：

```sql
-- 查看失败记录，找到原始 JSON 和错误原因
SELECT id, batch_id, error_message, created_at
FROM batch_upload_log
WHERE status = 'failed'
ORDER BY created_at DESC;
```

**排查步骤**：

1. 查看 `error_message` 确认错误原因
2. 从 `batch_upload_log.raw_json` 取出原始的 JSON 字符串
3. 修复问题后，重新调用 `CALL upsert_employee_tags(...)` 传入同样的 JSON
4. 存储过程会重新走一遍完整的"先存归档 → 解析转存"流程

**注意**：`batch_upload_log` 中每条记录都是独立的，同样的 `batch_id` 重跑多次会生成多条记录，各自有不同的 `id` 和 `status`。
