## 一、项目概述

**项目名称**：任职资格 AI 分析系统

**平台**：飞书 Aily（支持 Agent + 知识库 + 工作流 + MCP）

**目标**：对任职资格平台中每个人的关键成果和项目经历进行 AI 分析，生成：
1. **个人能力标签** — 按人员 + 岗位序列 + 原职级分类
2. **金标签** — 按岗位序列 + 职级 + 维度，标识区分通过/未通过的关键能力
3. **劣势诊断与精准提问** — 针对个人相对同岗位同职级人员的短板生成面试问题

**项目目的**：用户准备面试 AI 应用开发产品经理岗位，此项目作为面试作品展示专业能力。

## 二、术语约定（重要）

| 术语 | 说明 |
|------|------|
| **岗位序列** | 即之前说的"职系"。**用户要求：往后统一叫"岗位序列"**。如技术研发类、产品类、设计类等 |
| **原职级** | 即之前说的"原岗位"。员工当前职级，如 P5、P6 |
| **申报职级** | 员工本次评审申报的目标职级 |
| **金标签** | 同岗位序列 + 同职级中，能区分通过/未通过的关键能力标签 |
| **D 系数** | 区分度系数，D = (通过组标签均分 - 未通过组标签均分) / 未通过组标签均分 |

> **注意**：设计文档 HTML 中仍有大量"职系"字样未改为"岗位序列"，SQL 手册和智能体 Prompt 已全部更新。后续工作如涉及 HTML 修改，需统一替换。

## 三、业务逻辑

### 3.1 评审周期

- 每两年一次，员工自主申报目标岗位序列 + 职级
- 勾选项目经历和关键成果作为申报材料
- 评审委员会评审 → 通过 / 不通过

### 3.2 不同岗位序列的能力维度

| 岗位序列 | 能力维度 |
|---------|---------|
| 技术研发类 | 技术专长、技术创新、客户导向、知识技能 |
| 产品类 | 产品规划、用户洞察、数据分析、商业思维、项目管理 |
| 设计类 | 设计方法论、用户研究、技术实现、审美判断、行业视野 |
| 市场类 | 市场洞察、策略制定、创意能力、渠道管理、效果评估 |
| 管理类 | 团队管理、战略规划、资源协调、绩效驱动、人才培养 |

> 维度作为**可配置的元数据层**存储在数据库中，通过 MCP 工具 `get_dimension_schema` 动态获取，不写死在 Prompt 里。

### 3.3 个人能力标签提取逻辑

```
分组键 = (employee_id, position_family_code, original_grade)
```

- **同一个人、同一个岗位序列、同一个原职级**的材料合并为一组
- 只取最新一次评审记录，不跨周期合并
- 每条材料归入一个能力维度 → 每个维度下生成 1-3 个标签
- 每个标签包含：tag_name、score(0-100)、confidence(0-1)、evidence、source_materials

### 3.4 金标签生成逻辑

```
分组键 = (position_family_code, grade, dimension)
```

- 按岗位序列 + 职级分组（如"技术研发-P6"）
- 组内分为通过组(A)和未通过组(B)
- 在每个维度下逐标签计算区分度 D
- 判定条件：D > 0.25 且 通过组该标签均分 > 65 → 金标签

## 四、技术架构

### 双轨架构

| 轨道 | 触发方式 | 功能 |
|------|---------|------|
| **预生成 Pipeline** | 定时触发（每夜/每周） | 批量拉取数据 → AI 分析 → 生成标签/金标签 → 入库 |
| **HR 分析 Agent** | HR 实时对话 | 理解意图 → 调用 MCP 检索预生成数据 → 实时深度分析 → 输出报告 |

### 使用的 Prompt 工程技术

| 场景 | 技术 |
|------|------|
| 维度分类 | Few-shot + Structured Output |
| 能力标签生成 | Chain-of-Thought + Few-shot |
| 金标签识别 | Self-Consistency + Tree-of-Thought |
| 劣势诊断与提问 | ReAct + Structured Output |
| 整体 Agent | Plan-and-Execute |

## 六、MCP 工具设计

### 已实现的 MCP（用户已完成开发）

**`query_assessment_materials`** — 合并查询接口，一次性拉取评审活动下所有申报人员 + 关键成果 + 项目经历

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `rn_code` | string | **否** | 评审活动ID，不填则查所有评审活动 |
| `post_code` | string | 否 | 岗位序列代码 |
| `lev_code` | string | 否 | 职级代码 |
| `page_size` | int | 否 | 每页行数，默认 100 |
| `start` | int | 否 | 偏移量，第一页为 0 |

返回字段：assessment_activity_id, assessment_id, employee_id, employee_name, original_position, original_grade, position_family_code, position_family_name, target_grade, result, status, material_type(关键成果/项目经历), title, description, quantified_results, role, skill

## 七、SQL 查询接口设计

### 三个对外接口（SQL Server T-SQL）

| 接口 | 路径 | 用途 |
|------|------|------|
| 合并接口 | `GET /api/v1/assessments/materials-by-activity` | 一次性拉取人员+材料，UNION ALL 合并关键成果和项目经历 |
| 评审汇总接口 | `GET /api/v1/assessments/summary` | 返回某岗位序列+职级下谁过了谁没过，用于金标签 D 系数计算 |
| 增量更新接口 | `GET /api/v1/assessments/latest` | 拉取当天有变更的记录 |

**SQL 关键设计**：
- 分页：`OFFSET @start ROWS FETCH NEXT @PageSize ROWS ONLY`（SQL Server 2012+）
- 总行数：每个查询都先 `SELECT COUNT(*) AS total_rows`
- 可选筛选：`AND (@RNCode IS NULL OR R.RNCode = @RNCode)` 模式
- 字段返回英文 key + 中文 value（如 `material_type = N'关键成果'`）

### AI 分析系统接口（PostgreSQL）

**写入接口（PUT，幂等）**：
- `PUT /api/v1/tags/employee` — 写入个人能力标签（UPSERT）
- `PUT /api/v1/tags/gold` — 写入金标签（UPSERT）
- `PUT /api/v1/tags/benchmark` — 写入岗位基准（UPSERT）

**查询接口（GET）**：
- `GET /api/v1/tags/employee/{employee_id}` — 查个人标签
- `GET /api/v1/tags/gold/{position_family}/{grade}` — 查金标签
- `GET /api/v1/tags/benchmark/{position_family}/{grade}` — 查基准
- `GET /api/v1/tags/compare/{employee_id}` — 对标分析
- `GET /api/v1/tags/weakness/{employee_id}` — 劣势诊断

## 八、智能体 Prompt

### 已创建：标签生成 Agent Prompt

文件：`智能体Prompt-标签生成Agent.md`

**核心设计**：
- 系统角色：任职资格能力标签生成 Agent
- 调用 MCP `query_assessment_materials` 拉取数据
- 5 步工作流：
  1. 拉取数据（处理分页）
  2. 按 (employee_id, position_family_code, original_grade) 分组
  3. 维度分类（每条材料归入一个能力维度）
  4. 每个维度下生成 1-3 个标签
  5. 输出结构化 JSON
- 标签含：tag_name, score(0-100), confidence(0-1), evidence, source_materials
- 评分标准：90-100 卓越 / 80-89 优秀 / 70-79 良好 / 60-69 达标 / <60 待提升
- 5 条约束：只处理已完结评审、不保留历史版本、证据可追溯、维度映射唯一、批量输出

## 十二、关键设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 标签生成时机 | 预生成 + 定时更新 | 成本控制（批量复用上下文）、体验优化（秒级响应）、一致性保障 |
| 标签分组键 | 人员 + 岗位序列 + 原职级 | 同一人可能评审不同岗位序列；原职级代表能力基线 |
| 金标签分组键 | 岗位序列 + 职级 | 按岗位分组样本太小，按岗位序列+职级保证统计显著性 |
| 维度管理 | 可配置元数据（数据库存储） | 新增岗位序列只需更新配置，不改代码 |
| 接口合并 | 单一合并接口（UNION ALL） | Pipeline 不用循环调用，一次拉完 |
| 写入接口用 PUT | 幂等设计 | 标签数据可重复生成，PUT 天然幂等 |
| 字段命名 | 英文 key + 中文 value | 英文 key 便于程序处理，中文 value 便于 AI 理解 |
| 分页变量名 | `start`（非 `offset`） | 用户指定，避免与 SQL 关键字冲突 |
| 评审活动筛选 | 可选（非必填） | 支持全量查询和按活动查询两种模式 |
| SQL 方言 | SQL Server T-SQL | 用户的任职资格平台使用 SQL Server |
